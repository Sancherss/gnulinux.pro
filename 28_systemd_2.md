# Система инициализации systemd

## I. Ответы на контрольные вопросы

### 1. Операции с сервисами
Помимо стандартного запуска (`start`) и остановки (`stop`), ключевыми операциями являются:
* **Restart:** Полный перезапуск процесса (PID меняется). Используется при критических изменениях.
* **Reload:** Перечитывание конфигурации «на лету» без остановки процесса. **Дополнение:** Это критически важно для высоконагруженных сервисов (Nginx, SSH), так как позволяет избежать разрыва активных соединений.

### 2. Просмотр информации
* Основной инструмент — `systemctl status <service>`.
* **Дополнение:** Если `status` дает мало инфы, используем `journalctl -u <service>` для чтения системного журнала. Для получения «сырых» данных о юните используем `systemctl show`.

### 3. Различие Restart и Reload
* **Restart:** Процесс завершается и создается заново.
* **Reload:** Процессу посылается сигнал (SIGHUP) для обновления настроек.
* **Ключевое отличие:** При `restart` сервис временно недоступен, при `reload` — доступен непрерывно.

### 4. Запрет запуска (Mask vs Disable)
* **Disable:** Убирает из автозагрузки. Сервис можно запустить вручную.
* **Mask:** Создает симлинк на `/dev/null`. Сервис становится «невидимым».
* **Дополнение:** `mask` — это защитный механизм от «случайного» запуска сервиса другими зависимыми службами, чего `disable` гарантировать не может.

### 5. Поиск проблем
* Индикаторы в `status`: красный цвет, статус `failed`, ненулевой `exit code`.
* **Дополнение:** Важно проверять строку `Main PID` — если сервис постоянно перезапускается, его PID будет быстро меняться.

### 6. Статусы Running и Exited
* **active (running):** Сервис-демон, постоянно находящийся в оперативной памяти.
* **active (exited):** Сервис-скрипт. Выполнил задачу и завершился.
* **Дополнение:** Состояние `exited` — норма для таких юнитов, как настройка сети или установка параметров ядра.

### 7. Просмотр различных типов юнитов
* Команда: `systemctl list-units --type=<тип>`.
* **Примеры типов:** `service`, `target`, `mount`, `timer`, `socket`.

## II. Ответы на практические задания

В ходе практики были отработаны сценарии управления сервисом `sshd`.

### Задание 1-2: Управление автозагрузкой и состоянием
1.  **Действие:** Выключена автозагрузка (`systemctl disable sshd`).
2.  **Результат:** После перезагрузки в поле `Loaded` отобразилось `disabled`. Сервис не запустился автоматически.
3.  **Вывод:** Команда `disable` влияет только на старт при загрузке системы, но не останавливает уже запущенный процесс.

### Задание 3: Блокировка сервиса (Masking)
1.  **Действие:** Выполнена команда `systemctl mask sshd`.
2.  **Наблюдение:** Система создала символическую ссылку `/etc/systemd/system/sshd.service → /dev/null`.
3.  **Попытка запуска:** При команде `start` получена системная ошибка:
    > `Failed to start sshd.service: Unit sshd.service is masked.`
4.  **Восстановление:** После `unmask` сервис вернулся в состояние `loaded`, но остался выключенным (`inactive`).

### Задание 4: Финальная проверка
1.  **Действие:** Возвращение в автозагрузку (`enable`) и перезапуск.
2.  **Результат:** Статус `active (running)`, загрузка прошла успешно.
