# SELinux — Принудительный контроль доступа

## 1. Основная проблема, которую решает SELinux

**Стандартные права Linux (DAC — Discretionary Access Control)**

В обычном Linux права основаны на владельце файла:
- Если процесс запущен от пользователя `sancher` → ему доступны **все** файлы `sancher`
- Ядро не различает «хорошие» и «плохие» программы от одного пользователя

**Пример проблемы:**
> Firefox (браузер) и SSH-клиент — оба запущены от `sancher`. Оба имеют доступ к `~/.ssh/id_rsa`. Если Firefox взломают — злоумышленник украдёт SSH-ключ.

**Что нужно:** система, которая различает **программы** (даже от одного пользователя). Это и есть **принудительный контроль доступа (MAC)**.

---

## 2. Архитектура SELinux

SELinux встроен в ядро Linux через **фреймворк LSM** (Linux Security Module). SELinux не заменяет стандартные права, а **добавляется поверх них**.  
Сначала проверяются обычные `rwx`, потом — SELinux-политики.

---

## 3. Контекст безопасности (метка)

**Каждый объект в системе имеет метку (контекст):**
- Процессы
- Файлы
- Порты
- Устройства
- Пользователи (SELinux-пользователи)

### Формат контекста:
```
USER:ROLE:TYPE:LEVEL
```

### Пример:
```
unconfined_u:object_r:user_home_t:s0
```

### Компоненты контекста (по порядку):

| Место | Название | Для чего | Типичное значение |
|-------|----------|----------|-------------------|
| 1 | **User** | SELinux-пользователь | `unconfined_u`, `system_u`, `root` |
| 2 | **Role** | Роль (набор разрешений) | `object_r` (файлы), `system_r` (процессы) |
| 3 | **Type** | Тип (файл) / **Домен** (процесс) — **главное!** | `sshd_exec_t`, `user_home_t`, `etc_t` |
| 4 | **Level** | Уровень MLS (многоуровневая безопасность) | `s0` (по умолчанию) |

### Почему тип — главный?

Именно **тип** (или домен для процессов) участвует в правилах доступа.  
Пользователь и роль в стандартной конфигурации (targeted policy) **не влияют** на ограничения — они для сложных сценариев.

---

## 4. Как работает проверка доступа

### 4.1. Сущности и их домены/типы

| Сущность | Имеет контекст | Как называется "тип" | Пример |
|----------|---------------|---------------------|--------|
| Процесс | Да | **Домен** | `sshd_t`, `httpd_t`, `unconfined_t` |
| Файл | Да | **Тип** | `sshd_exec_t`, `shadow_t`, `etc_t` |
| Порт | Да | **Тип порта** | `ssh_port_t`, `http_port_t` |
| Устройство | Да | **Тип устройства** | `device_t` |

### 4.2. Принцип проверки

Когда процесс пытается выполнить действие (читать/писать/выполнять/слушать):

**Шаг 1:** Стандартные права Linux (владелец/группа/остальные)
- Если запрещено → дальше не идём
- Если разрешено → идём к шагу 2

**Шаг 2:** SELinux проверяет:
> *Может ли ДОМЕН процесса выполнить ДЕЙСТВИЕ над ТИПОМ объекта?*

- Действия: `read`, `write`, `execute`, `bind` (для портов), `transition` (запуск) и т.д.
- Если в политике есть разрешающее правило → доступ разрешён
- Если правила нет → доступ **запрещён** (даже если стандартные права разрешают)

### 4.3. Пример с SSH на нестандартном порту

| Что проверяет SELinux | Домен процесса | Тип порта | Разрешено? |
|----------------------|---------------|-----------|------------|
| Порт 22 | `sshd_t` | `ssh_port_t` | Да (правило есть) |
| Порт 2233 | `sshd_t` | `????????` | Нет (правила нет) |

Команда `semanage port -a -t ssh_port_t -p tcp 2233` **добавляет тип `ssh_port_t` к порту 2233**, после чего правило начинает работать.

## 5. типы, роли, уровни

### 5.1. Type Enforcement (TE) — главный механизм

**Тип/домен — это сердце SELinux.**

- Каждый процесс работает в своём **домене** (например, `sshd_t`, `httpd_t`, `unconfined_t`)
- Каждый файл имеет свой **тип** (например, `shadow_t`, `etc_t`, `user_home_t`)
- Правила определяют, какие **действия** разрешены между доменами и типами

**Аналогия:**  
В офисе у каждого сотрудника есть бейдж (домен), у каждой двери — замок (тип).  
SELinux — это турникет, который пропускает только если у бейджа есть доступ к замку.

### 5.2. Role-Based Access Control (RBAC) — роли (обычно отключена)

- Роль — это набор разрешённых доменов
- Пользователь SELinux может иметь несколько ролей
- По умолчанию в `targeted` политике роли не ограничивают

### 5.3. Multi-Level Security (MLS) — уровни (по умолчанию выключен)

**Военная модель безопасности:**

| Уровень | Обозначение |
|---------|-------------|
| Несекретно | `s0` |
| Секретно | `s1` |
| Совершенно секретно | `s2` |

**Правила MLS:**
- Процесс может **читать** объекты своего уровня и ниже
- Процесс может **писать** в объекты своего уровня и выше

**Почему выключен по умолчанию:**  
Слишком строг для обычных серверов и сложен в настройке.

## 6. Режимы работы SELinux

```
SELinux
   │
   ├── enforcing   → политики применяются, запрещённое блокируется
   ├── permissive  → политики НЕ применяются, но нарушения ЛОГИРУЮТСЯ
   └── disabled    → SELinux полностью выключен
```

**Зачем нужен `permissive`?**  
Позволяет увидеть, что будет заблокировано, без фактической блокировки.  
Полезен при написании политик и отладке.

## 7. Как SELinux влияет на администратора

### 7.1. Стандартные сценарии работы SELinux

| Вы меняете | Что проверяет SELinux |
|------------|----------------------|
| Порт демона | Тип порта должен соответствовать домену |
| Путь к файлам конфигурации | Тип нового пути должен совпадать с ожидаемым типом |
| Путь к логам | Аналогично |
| Путь к исполняемым файлам | Тип должен быть `_exec_t` для соответствующего домена |
| Включение новой функции | Может потребоваться булево переключатель |

### 7.2. Типичная проблема

> «Я изменил порт/путь/настройки, перезапустил сервис — и он не запускается. Почему?»

**Ответ:** SELinux блокирует, потому что вы сделали то, что не описано в политиках.

**Решение:**  
1. Посмотреть логи: `sudo journalctl -e | grep avc`
2. Часто там же есть подсказка с командой
3. Добавить новое правило через `semanage`

## 8. Почему SELinux не защитил SSH-ключ от Firefox?

В начале лекции был пример: Firefox (взломанный) читает `~/.ssh/id_rsa`, и SELinux это не блокирует.

**Почему?**

Потому что в стандартной конфигурации:
- И Firefox, и SSH-клиент работают в домене **`unconfined_t`**
- Домен `unconfined_t` имеет доступ практически ко всему
- SELinux в режиме `targeted` не ограничивает пользовательские процессы от того же пользователя

**Как это исправить?**  
Создать отдельный домен для Firefox и написать правила, запрещающие доступ к `user_home_t`.  
Но это сложно и часто ломает работу программ, поэтому по умолчанию так не делают.

> Компромисс между безопасностью и удобством: SELinux защищает системные службы (`sshd`, `httpd`), но не ограничивает жёстко пользовательские приложения.

---

## 9. Типы политик SELinux

| Политика | Описание |
|----------|----------|
| **targeted** (по умолчанию) | Ограничиваются только **сетевые демоны** (sshd, httpd, bind...). Пользовательские процессы — `unconfined_t` |
| **strict** | Ограничены **все** процессы. Редко используется |
| **mls** | Включает многоуровневую безопасность (военная модель) |

## 10. Команды для работы

### Информация о SELinux

```bash
getenforce              # enforcing/permissive/disabled
sestatus                # подробная информация
```

### Контексты

```bash
id -Z                   # контекст текущего пользователя
ps -Z                   # контексты процессов
ls -Z file              # контекст файла
sudo semanage login -l  # привязка системных пользователей к SELinux
```

### Работа с портами

```bash
sudo semanage port -l | grep ssh   # какие порты разрешены для SSH
sudo semanage port -a -t ssh_port_t -p tcp 2233  # добавить порт
```

### Работа с файлами и политиками

```bash
# Временное изменение (пропадёт после restorecon)
sudo chcon -t user_home_t /path/to/file

# Добавление в политику (навсегда)
sudo semanage fcontext -a -t sshd_exec_t /path/to/file

# Применение политики к файлу
sudo restorecon -v /path/to/file

# Рекурсивно для всей директории
sudo semanage fcontext -a -t httpd_log_t "/var/log/myapp(/.*)?"
sudo restorecon -rv /var/log/myapp
```

### Булевы переключатели

```bash
getsebool -a                     # все переключатели
semanage boolean -l              # с описаниями
sudo setsebool -P NAME on        # включить навсегда
```

### Логирование

```bash
sudo journalctl -e | grep avc    # ошибки SELinux
sudo ausearch -m avc -ts recent  # альтернативный просмотр
```

### Экспорт/импорт политик

```bash
sudo semanage export -f myconfig    # экспорт
sudo semanage import -f myconfig    # импорт
```

---

**Ключевая фраза, которая объясняет 90% проблем с SELinux:**

> *SELinux блокирует действие, потому что домен процесса не имеет права выполнять это действие над типом объекта. Решение — либо изменить тип объекта (если он неправильный), либо добавить правило в политику (если настройка нестандартная).*
