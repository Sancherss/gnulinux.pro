# Работа с сетью

## Часть 1. Теоретические вопросы

### 1. Что такое «сетевой мост» (bridge)?
**Ответ:** Сетевой мост — гипервизор выступает в роли свитча, виртуалка становится равноправным участником сети: получает IP из того же диапазона, доступна извне и сама ходит в сеть.

**Дополнение:** В отличие от NAT, хост НЕ является роутером. Виртуалка ведет себя как обычный физический ПК в локальной сети. DHCP-сервер (роутер) выдает ей IP напрямую. Соседние компьютеры видят виртуалку как отдельное устройство.

### 2. Что выведет команда `ip a`?
**Ответ:** Список всех сетевых интерфейсов с их состоянием (UP/DOWN), MAC-адресами, IPv4 и IPv6 адресами, broadcast-адресами.

**Дополнение:** Также показывает MTU, длительность аренды IP (valid_lft) и тип адреса (dynamic/static). `lo` — loopback, `enp0s3`/`eth0` — физические/виртуальные интерфейсы.

### 3. Для чего нужен loopback (lo) интерфейс?
**Ответ:** Чтобы компьютер мог общаться с самим собой (localhost). Виртуальный интерфейс с адресом 127.0.0.1, работает даже при отключенной физической сети.

**Дополнение:** Позволяет программам на одном ПК общаться через TCP/IP без выхода в реальную сеть. Используется для тестирования (ping 127.0.0.1 проверяет работу стека протоколов), для связи сервисов (БД → веб-сервер на той же машине). Всегда в состоянии UP.

### 4. Для чего используется команда `ping`? Принцип работы.
**Ответ:** Проверить доступность удаленного хоста, измерить задержку (RTT), по TTL оценить количество маршрутизаторов на пути.

**Принцип:** Отправляет ICMP Echo Request → удаленный хост отвечает ICMP Echo Reply. Замеряется время «запрос-ответ». TTL уменьшается на 1 на каждом маршрутизаторе. По разнице начального TTL (64/128/255) и полученного вычисляется число хопов.

### 5. Что произойдет с TTL при ping с 192.168.0.150 на 192.168.1.100?
**Ответ:** Так как адреса из разных подсетей (0.x и 1.x), пакет ОБЯЗАТЕЛЬНО пройдет через маршрутизатор. TTL уменьшится как минимум на 1 (а если маршрутизаторов несколько — на столько, сколько их).

**Дополнение:** Если бы адреса были в одной подсети (оба 192.168.0.x), маршрутизатор не нужен — пакет шел бы напрямую через ARP и switch, и TTL НЕ уменьшился бы.

### 6. Как отследить путь пакета?
**Ответ:** Командой `traceroute` (Linux/macOS) или `tracert` (Windows).

**Дополнение:** Принцип работы — отправка пакетов с TTL=1,2,3... Каждый промежуточный маршрутизатор при TTL=1 убивает пакет и сообщает о себе. Так строится цепочка IP-адресов с задержками до каждого узла. Звездочки (`*`) означают, что узел не отвечает (не баг, а политика безопасности).

### 7. Где посмотреть/изменить DNS? Можно ли использовать несколько?
**Ответ:** В файле `/etc/resolv.conf`. Можно использовать несколько DNS — они прописываются каждый с новой строки `nameserver IP`.

**Дополнение:** Система опрашивает DNS по порядку: первый не ответил — идет ко второму. В современных дистрибутивах файл часто генерируется автоматически (NetworkManager, systemd-resolved), поэтому лучше менять DNS через `nmtui` или `nmcli`, а не вручную.

### 8. Что выведет `nslookup google.com`?
**Ответ:** Показывает, к какому DNS-серверу мы обращались (Server / Address), и какие IP-адреса (IPv4 и IPv6) соответствуют домену.

**Дополнение:** Вывод содержит «Non-authoritative answer» — значит, сервер не является владельцем зоны, а дал ответ из кэша. Если DNS не работает — будет ошибка «can't find google.com».

## Часть 2. Практические задания

### Задание 1. Включение/выключение интерфейса командой `ip`
**Команды:**
```bash
sudo ip link set enp0s3 down   # выключить интерфейс
sudo ip link set enp0s3 up     # включить интерфейс
```
**Проверка:** `ip a` показывает state DOWN/UP.

### Задание 2. Ping сайта gnulinux.pro (ровно 4 запроса)
**Команда:**
```bash
ping -c 4 gnulinux.pro
```
**Ключ `-c`** — count, ограничивает количество пакетов. Без него ping идет бесконечно до Ctrl+C.

### Задание 3. Настройка DHCP через nmtui
**Действия:** `nmtui` → Edit connection → IPv4 CONFIGURATION выбрать `Automatic` (вместо Manual). Удалить вручную прописанные Addresses, Gateway, DNS.

**Результат:** IP, маска, шлюз, DNS получаются автоматически от роутера (DHCP-сервера). Адрес может меняться при переподключении.

### Задание 4. Настройка Host-only адаптера
**Выполнено:**
1. В VirtualBox: Файл → Менеджер хостовых сетей → Создать (сеть 192.168.56.0/24)
2. В настройках ВМ: Адаптер 2 → Включить → Тип Host-only → выбрать созданную сеть
3. В CentOS через `nmtui` настроен интерфейс `enp0s8`:
   - IPv4: Manual
   - Address: `192.168.56.10/24`
   - Gateway: `192.168.56.1`
   - DNS: `192.168.56.1`

**Результат:**
```
enp0s8: inet 192.168.56.10/24 scope global (статический, valid_lft forever)
```
**Проверка связи с хостом:**
```bash
ping 192.168.56.1   # 3 packets, 0% loss, time ~1.2 ms
```
**DNS в /etc/resolv.conf:**
```
nameserver 192.168.1.1
nameserver 192.168.56.1
```

## Итоговая конфигурация виртуальной машины

| Интерфейс | IP-адрес | Подсеть | Тип подключения | Gateway | Доступ |
|-----------|----------|---------|-----------------|---------|--------|
| `enp0s3` | 192.168.1.204/24 (DHCP) | 192.168.1.0/24 | Bridge | 192.168.1.1 | Интернет + локальная сеть |
| `enp0s8` | 192.168.56.10/24 (статический) | 192.168.56.0/24 | Host-only | 192.168.56.1 | Только общение с хостом |
