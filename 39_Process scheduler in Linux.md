# Планировщик процессов в Linux

## 1. Назначение планировщика

**Планировщик процессов** — компонент ядра ОС, который решает:
- Когда какой процесс должен быть обработан процессором
- Как долго процесс будет выполняться
- В каком порядке процессы получают CPU

**Основная задача:** разделение времени (time sharing) между процессами.

---

## 2. Приоритеты и вежливость (nice)

### 2.1. Обычные процессы (SCHED_OTHER / SCHED_NORMAL)

Для обычных процессов используется понятие **niceness** (вежливость):

| Параметр | Диапазон | Значение |
|----------|----------|----------|
| **Nice (вежливость)** | от -20 до +19 | Чем **выше** число, тем **вежливее** процесс (получает меньше CPU) |
| **Приоритет (PRI)** | от 0 до 39 (для обычных процессов) | Чем **ниже** число, тем **выше** приоритет |

**Формула:**
```
PRI = NI + 20
```

**Примеры:**

| nice | PRI | Что означает |
|------|-----|--------------|
| -20 (максимальный приоритет) | 0 | Самый важный обычный процесс |
| 0 (по умолчанию) | 20 | Стандартный приоритет |
| +19 (минимальный приоритет) | 39 | Самый "вежливый" (отдаёт CPU другим) |

### 2.2. Real-time процессы (RT)

Процессы реального времени имеют **более высокий приоритет**, чем все обычные процессы.

| Параметр | Диапазон | Значение |
|----------|----------|----------|
| **RT приоритет** | от 1 до 99 | Чем **выше** число, тем **выше** приоритет |
| **Итоговый приоритет ядра** | от -100 до -2 | Чем **ниже** число, тем **выше** приоритет |

**Формула:**
```
PRI = -1 - rt_priority
```

**Примеры:**

| RT приоритет | Итоговый PRI | Значение |
|--------------|--------------|----------|
| 99 (максимальный) | -100 | Наивысший приоритет в системе |
| 50 | -51 | Высокий приоритет |
| 1 (минимальный RT) | -2 | Самый низкий среди RT, но всё ещё выше обычных |

## 3. Политики планирования

### 3.1. Общая классификация

| Политика | Тип | Описание |
|----------|-----|----------|
| **SCHED_OTHER** (или SCHED_NORMAL) | Обычные процессы | Стандартная политика для пользовательских процессов, честное распределение CPU |
| **SCHED_FIFO** | Real-time (RT) | First In, First Out — процесс выполняется, пока не заблокируется |
| **SCHED_RR** | Real-time (RT) | Round Robin — как FIFO, но с ограничением по времени кванта |
| **SCHED_BATCH** | Обычные | Для пакетных задач (низкий приоритет) |
| **SCHED_IDLE** | Обычные | Только когда система полностью простаивает |

**Посмотреть список доступных политик:**
```bash
chrt -m
```

### 3.2. Сравнение SCHED_FIFO и SCHED_RR

| Характеристика | SCHED_FIFO | SCHED_RR |
|----------------|-----------|----------|
| Квант времени | Неограничен (пока процесс сам не освободит CPU) | Ограничен (обычно 100 мс) |
| Вытеснение по времени | Нет | Да (по истечении кванта идёт в конец очереди) |
| При блокировке | Идёт в конец очереди своего приоритета | Идёт в конец очереди своего приоритета |
| Вытеснение более приоритетным | Да (текущий процесс приостанавливается) | Да |

### 3.3. Поведение SCHED_FIFO (на примере)

**Исходная очередь (все с rt_priority = 50):**
```
[100 (выполняется)] → [101] → [102]
```

**Сценарий А: Процесс 100 добровольно блокируется (ждёт данные)**
- Уходит в **конец** очереди
- Результат: `[101 (выполняется)] → [102] → [100]`

**Сценарий Б: Появляется процесс 103 с более высоким приоритетом (60)**
- Процесс 103 немедленно вытесняет выполняющийся процесс
- Вытесненный процесс **остаётся в голове** очереди
- Результат: `103 (выполняется)`, очередь 50: `[101] → [102] → [100]`

### 3.4. Поведение SCHED_RR

- Аналогично FIFO, но добавлен **квант времени**
- Если процесс не завершился и не заблокировался за квант → вытесняется и идёт в конец очереди
- Обеспечивает справедливое распределение CPU между процессами одного приоритета

## 4. Ключевое различие между nice и RT-приоритетами

| Характеристика | nice (обычные процессы) | RT приоритет |
|----------------|------------------------|--------------|
| Диапазон | -20 … +19 | 1 … 99 |
| Влияние на приоритет | Чем **меньше** nice → тем **выше** приоритет | Чем **больше** RT prio → тем **выше** приоритет |
| Отношение к другим | Конкурируют между собой | **Всегда** выше любых обычных процессов |
| Инструмент управления | `nice`, `renice` | `chrt` |

**Запомните:**
- **RT-процессы** (FIFO/RR) всегда имеют приоритет **выше**, чем **любой** обычный процесс (даже с nice = -20)
- В htop и top RT-процессы отображаются с отрицательными значениями PRI (например, -51, -71)

## 5. Практическая работа с chrt

### 5.1. Основные команды

| Действие | Команда |
|----------|---------|
| Посмотреть политику и приоритет процесса | `chrt -p <PID>` |
| Установить SCHED_FIFO с приоритетом 50 | `sudo chrt -f -p 50 <PID>` |
| Установить SCHED_RR с приоритетом 70 | `sudo chrt -r -p 70 <PID>` |
| Установить SCHED_OTHER (обычный) | `sudo chrt -o -p 0 <PID>` |
| Список доступных политик | `chrt -m` |
| Запустить новый процесс с RT-политикой | `sudo chrt -f 50 ./program` |

### 5.2. Пример: изменение политики процесса bash

```bash
# Найти PID процесса bash
ps -ef | grep bash

# Установить SCHED_FIFO с приоритетом 50
sudo chrt -f -p 50 7255

# Проверить изменения
chrt -p 7255
# Вывод: pid 7255's current scheduling policy: SCHED_FIFO
#        pid 7255's current scheduling priority: 50

# Установить SCHED_RR с приоритетом 70
sudo chrt -r -p 70 7255

# Вернуть обычную политику
sudo chrt -o -p 0 7255
```

### 5.3. Что показывает htop

| Политика | RT приоритет | PRI в htop | Пояснение |
|----------|--------------|------------|-----------|
| OTHER | нет | 20 | nice=0 → 20+0=20 |
| FIFO | 50 | -51 | -1-50 = -51 |
| RR | 70 | -71 | -1-70 = -71 |

**Важно:** Чем **меньше** число в PRI, тем **выше** реальный приоритет процесса.

### 5.4. Наследование приоритета

- Дочерние процессы при `fork()` наследуют политику и приоритет родителя
- Пример: если bash имеет SCHED_FIFO 50, то запущенный из него htop тоже получит SCHED_FIFO 50

## 6. Когда использовать RT-политики

### 6.1. Сценарии применения

| Сфера | Пример | Зачем |
|-------|--------|-------|
| **Аудио/видео** | JACK, PipeWire | Низкая задержка (real-time audio) |
| **Финансы** | Торговые системы | Микросекундная детерминированность |
| **Промышленность** | Контроллеры | Строгие временные рамки |
| **Игры** | Игровые серверы | Уменьшение "лагов" |
| **Критичные сервисы** | SSH при 100% CPU | Сохранить отзывчивость |

### 6.2. Пример: повышение приоритета SSH

```bash
# Найти PID sshd
pidof sshd

# Повысить приоритет (SCHED_FIFO, приоритет 50)
sudo chrt -f -p 50 $(pidof sshd)
```

### 6.3. Важные предупреждения

**Опасность:** Не ставьте критическим системным процессам слишком высокий RT-приоритет!

**Плохая практика:**
```bash
sudo chrt -f 99 -p 1    # systemd — может заблокировать систему!
```

**Хорошая практика:**
```bash
sudo chrt -r 30 -p $(pidof my_audio_app)
```

**Почему опасно:** RT-процесс с приоритетом 99 может заблокировать важные системные прерывания, что приведёт к зависанию системы.

## 7. Сравнение nice и chrt

| Инструмент | Диапазон | Влияние | Когда использовать |
|------------|----------|---------|---------------------|
| `nice` / `renice` | -20 … +19 | Только среди обычных процессов | Повседневные задачи, общее управление приоритетами |
| `chrt` | 1 … 99 (RT) | Процесс становится выше всех обычных | Критичные задачи с жёсткими требованиями к задержкам |

**Пример сравнения:**
```bash
# nice — снижает приоритет (процесс становится вежливее)
nice -n 19 ./background_task

# chrt — поднимает в real-time (выше всех обычных)
sudo chrt -f 50 ./important_task
```

## 8. Полезные команды для мониторинга

| Команда | Что показывает |
|---------|----------------|
| `top` | PRI (приоритет), NI (nice), в режиме `f` можно добавить RTPRIO |
| `htop` | PRI, NI, визуальное отображение (цветовая индикация RT-процессов) |
| `ps -e -o pid,comm,policy,rtprio,pri,nice` | Таблица со всеми политиками и приоритетами |
| `chrt -p <PID>` | Политика и RT-приоритет конкретного процесса |

**Пример ps:**
```bash
ps -eo pid,comm,policy,rtprio,pri,nice | head -10
# policy: TS (SCHED_OTHER), FF (FIFO), RR (Round Robin)
```

| Тип | Приоритет ядра (PRI) | RT prio | nice | Пояснение |
|-----|---------------------|---------|------|-----------|
| **RT (FIFO/RR)** | -100 … -2 | 99 … 1 | — | Самые высокие приоритеты |

| **SCHED_OTHER** (мин приоритет) | 39 | — | +19 | Минимальный среди обычных |
| **SCHED_IDLE** | ~39+ | — | — | Самый низкий в системе |

## 10. Ключевые выводы

1. **Планировщик** — решает, кто и когда получает процессор
2. **nice** влияет только на обычные процессы (SCHED_OTHER)
3. **RT-процессы** (FIFO/RR) всегда важнее обычных
4. **chrt** изменяет политику и RT-приоритет
5. **FIFO** — работает до блокировки, **RR** — с квантом времени
6. В **htop/top** отрицательный PRI = RT-процесс (очень высокий приоритет)
7. Используйте RT-политики **осознанно** — можно "уронить" систему
