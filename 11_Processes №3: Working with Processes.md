# 16. Процессы №3: Работа с процессами
## Вопросы

### Вопрос 1: Что делает команда kill?

**Ответ:** команда `kill` отправляет сигналы процессам.

#### Что такое сигналы?

Сигналы — это способ общения с процессами. Всего в Linux около 60 различных сигналов.

```bash
# Посмотреть все сигналы
kill -l
```

#### Основные сигналы

| Сигнал | Номер | Что делает |
|--------|-------|------------|
| `SIGTERM` | 15 | вежливо просит завершиться (по умолчанию) |
| `SIGKILL` | 9 | принудительно убивает |
| `SIGSTOP` | 17,19,23 | останавливает (приостанавливает) процесс |
| `SIGCONT` | 18 | возобновляет остановленный процесс |
| `SIGHUP` | 1 | перечитать конфигурацию |

#### Синтаксис

```bash
kill -SIGTERM PID
kill -15 PID
kill PID              # SIGTERM по умолчанию

kill -SIGKILL PID
kill -9 PID

kill -SIGSTOP PID
kill -STOP PID
```

> **Важно:** `kill` отправляет сигнал.  
> `kill -9` действительно убивает (SIGKILL нельзя перехватить).

---

### Вопрос 2: Как лучше всего убивать процессы?

**Ответ:** лучше использовать `kill` без опций (SIGTERM, 15) — это мягкое завершение.

#### Сравнение SIGTERM (15) vs SIGKILL (9)

| | SIGTERM (15) | SIGKILL (9) |
|--|--------------|-------------|
| **Вежливость** | вежливо просит завершиться | грубо убивает мгновенно |
| **Время на завершение** | да (закрыть файлы, очистить ресурсы) | нет |
| **Риск повреждения данных** | низкий | высокий |
| **Процесс может перехватить** | да | нет |
| **Процесс может игнорировать** | да | нет |

#### Почему SIGTERM может не убить процесс

Некоторые процессы **игнорируют SIGTERM**:

| Тип процесса | Примеры | Почему |
|--------------|---------|--------|
| Графические приложения | gedit, firefox, chrome | могут иметь несохранённые данные |
| Демоны | nginx, apache | перехватывают для graceful shutdown |
| Отладчики | gdb, strace | чтобы не убить отлаживаемый процесс |
| Зависшие процессы | любой | просто не отвечают |

#### Правильный порядок действий

```bash
# 1. Самое вежливое — попросить завершиться
kill PID                # SIGTERM (15)

# 2. Если не реагирует через пару секунд — добавить жёсткости
sleep 2
kill -9 PID             # только если не завершился

# 3. Крайний случай — принудительно
kill -9 PID
```

### Задание: Эксперимент с убийством родительского процесса

**Задача:**
1. Запустить процесс в фоновом режиме (`gedit`)
2. Найти его родительский процесс
3. Убить родительский процесс
4. Проверить дочерние процессы и найти нового родителя

---

### Мой эксперимент: SIGTERM vs SIGKILL

#### Шаг 1: Запуск процесса

```bash
gedit &
[1] 35924
```

#### Шаг 2: Поиск PID и PPID

```bash
ps -ef | grep 35924
sancher 35924 34505 6 22:57 pts/2 00:00:00 gedit
#                    ↑ PPID = 34505
```

#### Шаг 3: Поиск родителя

```bash
ps -ef | grep 34505
sancher 34505 3761 0 22:42 pts/2 00:00:00 /usr/bin/bash
```

**Цепочка:** `gedit (35924)` → `bash (34505)` → `...`

---

#### Шаг 4: Попытка убить родителя SIGTERM (15)

```bash
kill 34505
ps -ef | grep 35924
sancher 35924 34505 1 22:57 pts/2 00:00:00 gedit
#                    ↑ PPID всё ещё 34505! Процесс не умер!
```

**Результат:** процесс не убился — он проигнорировал SIGTERM.

---

#### Шаг 5: Убийство родителя SIGKILL (9)

```bash
kill -9 34505
```

**Проверка ДО и ПОСЛЕ:**

| Процесс | ДО `kill -9` | ПОСЛЕ `kill -9` |
|---------|--------------|-----------------|
| bash (34505) | жив (PID 34505) | **мёртв** (нет в ps) |
| gedit (35924) | PPID = 34505 | PPID = **2734** (systemd) |
| gedit (35924) | жив | **жив** (усыновлён) |

```bash
# После kill -9
ps -ef | grep 35924
sancher 35924 2734 0 22:57 ? 00:00:00 gedit
#                   ↑ новый родитель (systemd)
```

---

### Что произошло на самом деле?

| Команда | Сигнал | Процесс | Результат |
|---------|--------|---------|-----------|
| `kill 34505` | SIGTERM (15) | проигнорировал | остался жив |
| `kill -9 34505` | SIGKILL (9) | не может игнорировать | убит |

### Дочерний процесс (gedit) после смерти родителя

- gedit **не умер**
- Был **усыновлён** `systemd` (PID 2734)
- Продолжает работать

### Как проверить нового родителя (точно)

```bash
# Через /proc (более точный способ)
cat /proc/35924/status | grep PPid
# PPid: 2734

# Или через pstree
pstree -p 35924
# systemd(2734)───gedit(35924)
```

---

## 📊 Выводы из эксперимента

1. **SIGTERM (15)** — вежливая просьба, процесс может проигнорировать
2. **SIGKILL (9)** — ядерный вариант, процесс не может игнорировать
3. Некоторые процессы (bash, графические приложения) могут игнорировать SIGTERM
4. **Дочерние процессы не умирают вместе с родителем**
5. **Сироты усыновляются `systemd`** (PID 1 или PID 2734 для --user)
6. `ps` может показывать старую информацию — для точности используй `/proc`

## 📊 Сравнительная таблица сигналов

| Сигнал | Номер | Можно перехватить? | Можно игнорировать? | Типичное поведение |
|--------|-------|-------------------|---------------------|-------------------|
| SIGTERM | 15 | да | да | мягкое завершение |
| SIGKILL | 9 | нет | нет | мгновенное убийство |
| SIGINT | 2 | да | да | Ctrl+C (обычно завершает) |
| SIGSTOP | 19 | нет | нет | остановка процесса |
| SIGCONT | 18 | да | нет | возобновление |
| SIGHUP | 1 | да | да | перечитать конфиг |

### Основные команды

```bash
# Посмотреть все сигналы
kill -l

# Отправить сигнал
kill -SIGTERM PID
kill -15 PID
kill PID

# Принудительно убить
kill -9 PID

# Остановить процесс
kill -STOP PID

# Возобновить процесс
kill -CONT PID

# Информация о сигналах
man 7 signal
```

### Проверка статуса процесса

```bash
# Статус процесса (R, S, T, Z)
ps -o pid,stat,cmd -p PID

# Какие сигналы игнорирует/перехватывает процесс
cat /proc/PID/status | grep -E "SigIgn|SigCgt"

# Найти нового родителя (точный способ)
cat /proc/PID/status | grep PPid
```

### Когда какой сигнал использовать

| Ситуация | Сигнал |
|----------|--------|
| Обычное завершение программы | SIGTERM (15) |
| Программа не реагирует на SIGTERM | SIGKILL (9) |
| Временно приостановить процесс | SIGSTOP (19) |
| Возобновить процесс | SIGCONT (18) |
| Перечитать конфигурацию демона | SIGHUP (1) |
