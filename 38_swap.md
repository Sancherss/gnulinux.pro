# Виртуальная память и swap в Linux

## 1. Основные понятия

### 1.1. Виртуальная память
**Виртуальная память** = оперативная память (RAM) + swap-пространство на диске.

**Назначение:**
- Позволяет процессам использовать адресное пространство, превышающее физическую RAM
- Обеспечивает изоляцию процессов друг от друга
- Позволяет выгружать редко используемые страницы на диск

### 1.2. Буфер и кэш страниц

| Компонент | Что хранит | Характер данных |
|-----------|-----------|-----------------|
| **Буфер (buffer)** | Метаданные файловой системы, соответствие блоков диска страницам памяти | Служебные |
| **Кэш страниц (page cache)** | Содержимое файлов (страницы) | Пользовательские данные |

**Зачем нужен кэш:**
- При повторном чтении одного файла данные берутся из RAM (быстро)
- При частой записи не требуется каждый раз обращаться к диску

### 1.3. Dirty страницы (грязные страницы)
- Страницы в кэше, которые были изменены, но ещё не записаны на диск
- Помечаются как **Dirty**
- Периодически сбрасываются на диск (обычно раз в 5 секунд)

**Режимы кэширования при записи:**

| Режим | Поведение | Риск при отключении питания |
|-------|-----------|----------------------------|
| **Writeback** (по умолчанию) | Данные сначала в кэш, на диск — асинхронно | Есть (потеря dirty страниц) |
| **Writethrough** | Данные одновременно в кэш и на диск | Нет (медленнее) |

**Команда принудительного сброса:** `sync`

## 2. Типы страниц памяти

### 2.1. Классификация по двум осям

**Ось 1: Связь с диском**
- **File-backed** — страница связана с файлом на диске (можно выгрузить без swap)
- **Anonymous** — страница не связана с файлом (стек, куча, данные malloc)

**Ось 2: Доступность**
- **Private** — доступна только одному процессу
- **Shared** — доступна нескольким процессам

### 2.2. Итоговая матрица (4 типа)

| | **Private** | **Shared** |
|--|-------------|------------|
| **File-backed** | Исполняемый код, COW-копии при записи | Разделяемые библиотеки (.so), `mmap` с `MAP_SHARED` |
| **Anonymous** | Стек, куча (heap), переменные | POSIX shared memory, `mmap` с `MAP_SHARED` без файла |

### 2.3. Поведение при нехватке памяти

| Тип | Можно ли выгрузить без swap |
|-----|----------------------------|
| **File-backed** | Да — данные есть на диске |
| **Anonymous** | Нет — данных на диске нет → нужен swap |

## 3. Swap (подкачка)

### 3.1. Определение
**Swap** — место на диске (раздел или файл) для временного хранения выгруженных anonymous страниц.

### 3.2. Назначение swap
1. **Расширение виртуальной памяти** — запуск процессов с суммарным RSS больше RAM
2. **Освобождение RAM под page cache** — вместо хранения редко используемых anonymous страниц
3. **Гибернация (suspend-to-disk)** — сохранение всего содержимого RAM в swap

### 3.3. Важные уточнения
- Swap **не является продолжением оперативной памяти** — это временное хранилище
- При нехватке RAM иногда выгоднее выгрузить редко используемые страницы в swap, чтобы освободить место под кэш

### 3.4. Swap-раздел vs swap-файл

| Критерий | Раздел | Файл |
|----------|--------|------|
| Производительность (ядро 2.6+) | Одинаковая | Одинаковая |
| Увеличение размера | Сложно (если не LVM) | Просто |
| Перемещение | Сложно | Просто |
| Гибернация | Да | Да (с параметром resume=) |

### 3.5. Приоритеты swap (при наличии нескольких областей)
- У каждого swap-устройства/файла есть приоритет (чем выше число, тем выше)
- При одинаковом приоритете используется **round-robin**
- Сначала заполняются области с более высоким приоритетом

## 4. Параметры и настройка

### 4.1. swappiness (0–100)

**Что это:** параметр, определяющий баланс между выгрузкой anonymous страниц и удержанием page cache в RAM.

| Значение | Поведение | Когда применять |
|----------|-----------|-----------------|
| **0–30** | Swap редко, anonymous страницы держатся в RAM как можно дольше | Серверы БД (сами управляют кэшем) |
| **60** | Значение по умолчанию | Универсальные системы |
| **80–100** | Агрессивная подкачка, RAM освобождается под кэш | Файловые серверы, где кэш важнее |

### 4.2. Zswap (дополнительный модуль)
- Создаёт **сжатую область в RAM** для хранения swap-страниц
- Уменьшает количество операций записи на диск
- При заполнении — данные выгружаются на диск (обычный swap)

| Механизм | Хранение | Скорость | Износ диска |
|----------|----------|----------|-------------|
| Swap (диск) | Диск | Медленно | Есть |
| Zswap | RAM (сжатая) | Быстро | Нет (до заполнения) |

## 5. Swap на SSD: оценка рисков

### 5.1. Факты
- Современный SSD (250 ГБ, TLC) имеет ресурс **150–360 TBW** (терабайт записи)
- Активная подкачка (30 ГБ/день) → ресурс ~13 лет
- Реальная подкачка на системе с достаточной RAM (< 2 ГБ/день) → ресурс 50+ лет

### 5.2. Вывод
Swap на SSD **экономически оправдан** для большинства сценариев.

**Исключения:**
- Очень старые / дешёвые SSD (низкий ресурс)
- Системы с хронической нехваткой RAM (требуется увеличение RAM вместо swap на SSD)

## 6. Рекомендации по размеру swap

| RAM | Без гибернации | С гибернацией |
|-----|---------------|---------------|
| < 2 ГБ | ×2 от RAM | ×3 от RAM |
| 2–8 ГБ | = RAM | ×2 от RAM |
| 8–64 ГБ | 4–8 ГБ | ×1.5 от RAM |
| > 64 ГБ | 4 ГБ или 0 (по желанию) | 8–16 ГБ |

## 7. Команды для работы с памятью и swap

### 7.1. Мониторинг

| Команда | Назначение |
|---------|-----------|
| `free -h` | Просмотр RAM и swap (human-readable) |
| `free -m` | Просмотр в мегабайтах |
| `swapon --show` | Список активных swap-областей |
| `swapon -s` | То же (старый синтаксис) |
| `cat /proc/swaps` | Аналог swapon --show |
| `grep Dirty /proc/meminfo` | Размер грязных страниц |
| `sysctl vm.swappiness` | Текущее значение swappiness |
| `vmstat 1` | Мониторинг подкачки в реальном времени (колонки si/so) |
| `blkid \| grep swap` | Поиск swap-разделов |

### 7.2. Создание swap-файла

```bash
# 1. Создать файл (2 ГБ)
dd if=/dev/zero of=/swapfile bs=1K count=2M

# Альтернатива (быстрее)
fallocate -l 2G /swapfile

# 2. Установить права (обязательно!)
sudo chmod 600 /swapfile

# 3. Форматировать под swap
sudo mkswap /swapfile

# 4. Активировать с приоритетом (опционально)
sudo swapon -p 1 /swapfile

# 5. Проверить
swapon --show
```

### 7.3. Создание swap-раздела (через fdisk)

```bash
# 1. Создать раздел
sudo fdisk /dev/sdb
# n (новый раздел) → Enter → Enter → +2G (размер)
# t → 82 (тип Linux swap)
# w (записать)

# 2. Форматировать как swap
sudo mkswap /dev/sdb1

# 3. Активировать
sudo swapon /dev/sdb1
```

### 7.4. Управление swap

| Действие | Команда |
|----------|---------|
| Активировать конкретный swap | `swapon /путь` |
| Деактивировать | `swapoff /путь` |
| Деактивировать всё | `swapoff -a` |
| Активировать всё из /etc/fstab | `swapon -a` |

### 7.5. Постоянное подключение (/etc/fstab)

**Для swap-файла:**
```
/swapfile none swap sw 0 0
```

**Для swap-раздела (по UUID — рекомендуется):**
```
UUID=1234-5678-90ab-cdef none swap sw 0 0
```

**Для swap-раздела (по устройству):**
```
/dev/sdb1 none swap sw 0 0
```

### 7.6. Настройка swappiness

```bash
# Временное изменение
sudo sysctl vm.swappiness=30

# Постоянное изменение (создать файл)
echo "vm.swappiness=30" | sudo tee /etc/sysctl.d/99-swappiness.conf

# Применить изменения
sudo sysctl -p /etc/sysctl.d/99-swappiness.conf
```

## 8. Практический пример: тест dirty страниц

```bash
# 1. Сбросить dirty страницы на диск
sync

# 2. Проверить текущее значение Dirty
grep Dirty /proc/meminfo

# 3. Создать файл размером 100 МБ
dd if=/dev/urandom of=testfile bs=1M count=100

# 4. Снова проверить Dirty (должен показать ~100 МБ)
grep Dirty /proc/meminfo

# 5. Сбросить на диск
sync

# 6. Проверить Dirty (должно стать 0)
grep Dirty /proc/meminfo
```

**Вывод:** файл сначала создался в page cache (dirty), затем после `sync` записался на диск.

---

## 9. Ключевые выводы

| Понятие | Суть |
|---------|------|
| **Page cache** | Страницы файлов в RAM (ускоряет чтение/запись) |
| **Dirty pages** | Изменённые страницы, ожидающие записи на диск |
| **Anonymous memory** | Память без файла (стек, куча) — требует swap |
| **File-backed memory** | Память, связанная с файлом — можно выгрузить без swap |
| **Swap** | Место на диске для выгрузки anonymous страниц |
| **swappiness** | Баланс между удержанием anonymous страниц и кэша в RAM |
| **sync** | Принудительный сброс dirty страниц на диск |

## 10. Ответы на частые вопросы

### 10.1. Нужен ли swap, если много RAM?
Если RAM с большим излишком — можно без swap. Но swap может быть полезен для большего пространства под кэш.

### 10.2. Стоит ли делать swap на SSD?
Да, при достаточном объёме RAM износ будет минимальным. Скорость подкачки значительно выше, чем на HDD.

### 10.3. Раздел или файл подкачки?
На современных системах разницы в производительности нет. Файл удобнее в управлении (легче изменить размер).

### 10.4. Что лучше: выгружать кэш или использовать swap?
Зависит от сценария:
- **Низкий swappiness** — держать anonymous страницы в RAM, чаще выгружать кэш (СУБД)
- **Высокий swappiness** — держать кэш в RAM, чаще выгружать anonymous страницы (файловые серверы)
